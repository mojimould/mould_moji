\tPart[lot]{幾何学・運動学的特性の解析計算}{概要}{%
\paragraph*{目的}
モールドや工具等の幾何学的特性およびマシニング内での運動学的制約を考慮し、加工の際に必要となる位置・距離等の\textbf{解析的な導出}を試みる。
\tcbline*
\paragraph*{背景}
\textbf{ソフトウェアの観点}から見た場合、高精度な加工を実現するためにはモールドや工具の\index{きがかくてきとくせい@幾何学的特性}\textbf{幾何学的特性}、さらにはマシニング内での\index{うんどうがくてきとくせい@運動学的特性}\textbf{運動学的特性}を正確に把握することが不可欠であることは言うまでもない。
マシニングセンタの操作・指示はこうした特性の理解が前提であることは``常識"という言葉を用いても過言ではない
%% footnote %%%%%%%%%%%%%%%%%%%%%
\footnote{さらに、\index{CAD}CAD, \index{CAM}CAMを利用するというのであれば、これに加えてCADによる\index{3Dモデリング}3Dモデリングの技術、およびそれに伴うCAMの操作技術も習熟しなければならず、教育コストが大きくかかることは容易に推測される。}。
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

　モールドの場合、これらの特性はほとんどが直線または円で記述されるものであり、その体系化は（複雑ではあるが）難しいものではない。
しかし現時点（\DMname 設置時点）において、こうした特性の\textbf{体系化はほぼ全くなされていない}。
実際、\textbf{明細ごとに対応}をしている状態にある。

　したがって、（少なくとも実際の加工に関わる程度に）これらの\textbf{特性の把握・体系化}は急務である。
またこれにより品質や生産性の低下が大きく抑えられることが期待される。
\tcbline*
\paragraph*{結論}
\DMname のおける加工の際に必要なすべての幾何的情報を解析的に導出した。
}

\tPart{解析計算に基づく数値解析}{概要}{%
\paragraph*{目的}
前段階で導出した\textbf{解析的な情報}を用いて、各明細における具体的な\textbf{数値的な情報}に変換するシステムの構築を試みる。
\tcbline*
\paragraph*{背景}
一般に、個々の製品や工具等の寸法は異なり、明細ごとに\textbf{固有の寸法}等を持つ。
そのため実際の加工（プログラムの作成）の際には、それら全てに対して\textbf{具体的な数値情報}を指定する必要がある。

　こうした数値情報は明細ごとに膨大にあるが、現時点（\DMname 設置時点）において、こうした手続きは\textbf{明細ごとに手作業}で行われている
%% footnote %%%%%%%%%%%%%%%%%%%%%
\footnote{さらには、「\index{ずめん@図面}図面の作成後、それを見てプログラムを作成する」という明らかに異常な（奇妙な）事態が放置され続けている。}。
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

　したがって、こうした手続きの\textbf{システム化}を行い、可能な限り\textbf{自動化}することが喫緊の課題である。
そうすることで、\textbf{危険を伴う作業の削減（安全性の向上）}や、\textbf{品質の低下の防止}に大きく寄与できることが期待さされる。
また副次的効果として、作業効率・人的資源・安全(security)・保守など、いずれの観点からみた\textbf{能率（生産性）の低下の防止}にも大きく貢献することも自ずと期待される。
\tcbline*
\paragraph*{結論}
各明細のモールドにおける固有数値情報の入力により、加工の際に必要となる具体的な数値情報が自動的に得られるシステムを構築した。
}
\input{./RfCPN_part_Numerical_Calculation}

\tPart[lot]{諸規程の策定}{概要}{%
\paragraph*{目的}
\DMname についての（特にソフトウェアの観点による）\textbf{規程の策定}を試みる。
\tcbline*
\paragraph*{背景}
すべての業務において、何の決まりごともなく\textbf{管理状態}におくことは不可能である。

　しかし現時点（\DMname 設置時点）において、「ソフトウェアエンジニアリングに関する管理・業務を行う部門」は存在しておらず、専任の担当者さえも存在しない。
そのため、\textbf{ソフトウェアエンジニアリング視点による規程}等も存在しない
%% footnote %%%%%%%%%%%%%%%%%%%%%
\footnote{「ソフトウェア」を「ハードウェア」に置き換えてみると、これがどれほどの異常事態であるかがわかる。}。
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

　したがって、（\DMname 関連に限らず）こうした視点による規程を設けることが焦眉の課題である。
\tcbline*
\paragraph*{結論}
\DMname 関連のソフトウェア開発における、（ソフトウェアの観点による）寸法・コードの記述法・工具・保守等の規程を策定した。
}

\tPart[lot]{諸標準の策定：G-codeの記述}{概要}{%
\paragraph*{目的}
\DMname についての（特にソフトウェアの観点による）\textbf{標準の策定}を試みる。
\tcbline*
\paragraph*{背景}
G-codeの記述に関しては、その規格が多岐にわたり統一された標準が存在しない。
そのため、G-codeの記述に関する社内における標準を参照する必要がある。

しかし、現時点(\DMname 設置時点)において、G-codeの記述に関する社内の標準は一切存在しない。

　したがって、G-codeの記述に関する標準の策定が急務である。
これにより、G-codeの記述における一貫性と効率性が向上し、業務の効率化（外注を含む）とコスト削減が期待される。
また標準を設けることにより、G-codeの記述に関する混乱を解消し、より高品質なソフトウェア開発を実現するための重要なステップとなる。
\tcbline*
\paragraph*{結論}
\DMname 関連のソフトウェア開発における、（ソフトウェアの観点による）寸法・コードの記述法・工具・保守等の標準を策定した。
}
\input{./RfCPN_part_Standards_Coding}
